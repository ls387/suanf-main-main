# 📋 排课管理系统前端建设规划书（完整版）

**目标**：将现有排课算法系统升级为完整的排课管理平台

---

## 📊 第一章 需求总览

### 1.1 用户痛点与解决方案

| 痛点                 | 后果                       | 解决方案                         |
| -------------------- | -------------------------- | -------------------------------- |
| 无实时排课进度显示   | 容易连续点击排课，系统混乱 | P0-1: 排课版本管理+WebSocket进度 |
| 排课结果无法直观验证 | 冲突问题隐藏，难以发现     | P1-1: 冲突检测可视化+课表展示    |
| 约束权重不可调       | 排课结果难以满足不同需求   | 【新】需求1: 约束权重动态调节    |
| 排课后无法手动微调   | 小问题无法快速修复         | 【新】需求3: 手动拖拽调课        |
| 版本管理不规范       | 难以追溯和对比方案         | 【新】需求+版本链管理            |
| 基础数据无法编辑     | 排课质量受限               | P1-4: 数据编辑管理系统           |
| 无数据流转能力       | 难以与其他系统集成         | P2-1: 导入导出系统               |
| 教务无法实时答疑     | 排课决策效率低             | P2-2: AI智能助手                 |

### 1.2 功能需求总表

**共11个核心功能模块**，分4个优先级：

| 优先级            | 功能                                 | 核心价值                           |
| ----------------- | ------------------------------------ | ---------------------------------- |
| **P0** ⭐⭐⭐⭐⭐ | **排课版本管理**                     | 【必须】排课管理基础，解决连点问题 |

| **P0** ⭐⭐⭐⭐⭐ | **【新】冲突检测接口（可以先不弄）** | 【必须】拖拽交互基础               |
| **P1** ⭐⭐⭐⭐   | **冲突检测可视化页面**               | 排课质量可视化                     |

| **P1** ⭐⭐⭐     | **多视角课表**                       | 满足不同角色需求                   |

| **P2** ⭐⭐⭐⭐   | **【新】手动拖拽调课**               | 排课精细化调整                     |

## 📈 第二章 优先级划分依据

### 2.1 P0级 - 基础架构

#### P0-1: 排课版本管理系统

**为什么最重要**：

- ✅ 是整个系统的基础，其他功能都依赖版本数据
- ✅ 解决"用户无法看排课进度"的痛点
- ✅ 防止用户连续点击导致系统混乱
- ✅ 所有排课结果都要有版本追踪

**核心功能**：

1. 前端动态创建版本（学期、年级、计划名）→ 后端分配version_id
2. WebSocket实时进度显示（进度条、代数、适应度、预计时间）
3. 排课完成后显示成效报告（冲突数、覆盖率、利用率）
4. 方案确认、导出Excel、版本列表查看

**后端改造**：

- 改造 `POST /api/v1/scheduling/run` 为异步+任务队列
- 新增 `WS /ws/scheduling/{version_id}` WebSocket端点
- 新增 `POST /api/v1/scheduling/{version_id}/confirm` 确认API
- 新增 `GET /api/v1/scheduling/{version_id}/export` 导出API

**前端新增**：

- `VersionCreate.vue` - 版本创建表单
- `SchedulingProgress.vue` - 实时进度显示
- `VersionList.vue` - 版本列表

**验收标准**：

- ✅ 版本创建后能动态分配ID
- ✅ 排课时实时显示5个关键指标，更新频率≤3秒
- ✅ 排课完成后显示报告，可确认/导出
- ✅ 防止重复点击（排课中禁用按钮）

---

#### 【新】需求1：约束权重动态调节

**为什么重要**：

- ✅ 让教务老师灵活控制排课倾向（尊重教师偏好 vs 最大化教室利用率）
- ✅ 不同学期/年级有不同排课策略
- ✅ 提供预设配置，减少学习成本

**核心功能**：

1. 在版本创建时显示【高级设置】折叠面板
2. 支持8项硬约束 + 10项软约束权重微调
3. 5个系统预设（快速/均衡/高质量/尽可能约束/宽松）
4. 支持自定义预设保存

**后端改造**：

- 新增 `ConstraintWeights` 数据模型
- 新增 `scheduling_constraint_presets` 数据库表（5条系统预设）
- 改造 `SchedulingRequest` 支持 `constraint_weights` 参数
- 在 `SchedulingService.run_scheduling()` 中应用权重

**前端新增**：

- `ConstraintWeightsPanel.vue` - 权重配置面板
- 预设管理页面
- 权重验证逻辑

**验收标准**：

- ✅ 权重调整后排课结果有明显变化
- ✅ 预设一键切换，无需手动输入
- ✅ 版本记录使用的权重快照
- ✅ 权重范围有效提示

---

#### 【新】需求2：冲突检测接口（仅检测不排课）

**为什么重要**：

- ✅ 为手动拖拽课程提供实时冲突反馈
- ✅ 不需要重新排课，快速反应
- ✅ 支持"冲突修复建议"

**核心功能**：

1. 新API：`POST /api/v1/scheduling/conflict-check`
2. 输入：版本ID 或 临时课表数据
3. 输出：冲突列表 + 统计 + 修复建议
4. 支持4类冲突分类（教师/班级/教室/容量）

**后端改造**：

- 复用 `SchedulingGeneticAlgorithm` 的冲突检查函数
- 新增 `ConflictCheckService.check_conflicts_only()` 方法
- 新增修复建议生成器（可选LLM或规则引擎）

**前端新增**：

- 调用冲突检测的工具函数
- 冲突反馈组件

**验收标准**：

- ✅ API响应时间 < 500ms
- ✅ 冲突检测准确率 = 排课算法中的检查逻辑
- ✅ 支持多次调用（用于拖拽交互）

### 2.3 P2级 - 增强功能

#### 【新】需求3：手动拖拽调课 + 版本链

**为什么重要**：

- ✅ 排课完成后仍可精细化调整
- ✅ 小问题无需重新排课
- ✅ 版本链记录所有修改（可回溯）

**核心功能**：

1. **编辑模式**
   - 点击【进入编辑模式】进入课表编辑
   - 课程卡片变为可拖拽
   - 拖拽反馈：显示可能的冲突

2. **拖拽交互**
   - **水平拖拽**：改变星期（周一→周二）
   - **垂直拖拽**：改变时间（9-10节→11-12节）
   - **拖到教室列表**：改变教室
   - 每次拖拽后实时调用冲突检测接口，显示当前冲突

3. **编辑工具栏**
   - [撤销] [重做] [重置]：编辑历史管理
   - [预览对比]：对比修改前后
   - [保存为版本]：保存修改为新版本

4. **版本链**
   - 修改保存时创建新版本（parent_version_id = 原版本）
   - 支持版本树（一个版本可有多个子版本）

- 改造 `schedule_versions` 表：添加 `parent_version_id`, `version_type`, `adjustment_name`, `changes_summary` 字段

## 🎯 第四章 关键决策点

### 4.1 关键决策

| 决策点           | 选项                       | 建议      | 理由                     |
| ---------------- | -------------------------- | --------- | ------------------------ |
| **课表实现技术** | CSS Grid vs Flexbox vs DIV | CSS Grid  | 自动跨行，天生适合日历表 |
| **拖拽库**       | vue-draggable vs @dnd-kit  | @dnd-kit  | Vue3原生支持，性能更好   |
| **排课实时性**   | 轮询 vs WebSocket          | WebSocket | 低延迟，服务器推送友好   |
| **权重存储**     | 数据库 vs 内存缓存         | 数据库    | 需要版本追踪，保证一致性 |
| **AI集成方式**   | 实时调用 vs 后台队列       | 实时调用  | 用户体验要求，可加缓存   |
| **版本链设计**   | 树形 vs 链表               | 树形      | 支持多分支调整，更灵活   |
| **冲突检测**     | 前端 vs 后端               | 后端+缓存 | 准确性优先，后端计算     |

### 4.2 风险与缓解

| 风险                | 影响         | 可能性 | 缓解方案                          |
| ------------------- | ------------ | ------ | --------------------------------- |
| 课表网格复杂度      | P1-2延期     | 中     | 提前做POC，明确CSS Grid用法       |
| WebSocket连接稳定性 | P0-1延期     | 中     | 使用成熟库+心跳检测+自动重连      |
| 拖拽性能问题        | P2-3延期     | 中     | 虚拟滚动、防抖、按需渲染          |
| AI回复准确性        | P2-2难以验收 | 高     | 提前与neiqian团队沟通，建立测试集 |
| 数据库表结构变更    | 数据迁移失败 | 低     | 完整的迁移脚本+回滚方案           |
| 与现有系统兼容性    | 无法集成     | 低     | 前期做完整的技术调研              |
