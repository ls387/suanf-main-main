

你是一个高级软件开发专家，擅长构建基于大语言模型（LLM）的智能对话系统。请根据以下详细需求，生成一个完整的、可运行的后端服务代码。

---

#### **项目概述**
开发一个高校教务智能问答助手，作为 API 服务嵌入网站。用户可以通过文本聊天询问课程安排、教室占用、教师信息等教务相关问题，助手能理解自然语言，查询 MySQL 数据库，并返回准确答案。系统需支持多轮对话、主动反问以收集缺失信息，并保证只读操作和安全性。

---

#### **功能需求**
1. **接收用户消息**：通过 API 接收用户输入的文本和会话 ID。
2. **多轮对话管理**：维护每个会话的最近 N 轮对话历史（例如 5 轮）。
3. **理解用户意图**：结合对话历史和当前时间（如“今天下午3点”），判断是否需要查询数据库。
4. **主动反问**：如果信息不足，生成追问问题，引导用户提供更多信息。
5. **生成 SQL 查询**：当需要查询时，让 LLM 生成一条只读的 SELECT SQL 语句，并用 `<sql>` 标签包裹。
6. **执行 SQL**：提取 SQL，使用只读数据库账户执行，获取结果。
7. **生成最终回答**：将查询结果和用户原始问题再次提供给 LLM，生成自然语言回答。
8. **安全限制**：
   - 仅允许 SELECT 查询，禁止其他操作。
   - 使用参数化查询或安全的 SQL 构建方式，防止注入。
   - 限制查询返回行数（如 100 行）。
   - 设置查询超时。
9. **错误处理**：对无效 SQL、数据库连接失败、LLM 调用失败等情况进行优雅处理，返回友好的错误消息。

---

#### **技术栈要求**
- **语言与框架**：Python + FastAPI（或 Node.js + Express，优先 Python）
- **数据库**：MySQL（使用 `mysql-connector-python` 或 `aiomysql` 异步驱动）
- **LLM 集成**：假设使用 OpenAI GPT 模型/deepseek/千问（或兼容 API），通过 HTTP 请求调用。需配置 API 密钥和模型名称（如 `gpt-4`）。
- **会话存储**：使用内存字典（简单实现，生产环境建议用 Redis）存储对话历史。

---

#### **API 设计**
- **端点**：`POST /chat`
- **请求体 JSON**：
```json
{
  "session_id": "string",  // 会话标识，新会话可传空或新 UUID
  "message": "string"      // 用户输入的文本
}
```
- **响应体 JSON**：
```json
{
  "session_id": "string",
  "reply": "string",       // 助手的回复文本
  "requires_sql": false    // 可选，指示是否需要前端处理 SQL（本系统内部处理，可省略）
}
```
- **内部流程**：
  1. 根据 `session_id` 获取或创建对话历史。
  2. 构建系统提示词（包含数据库 Schema、时间信息、Agent 角色等）。
  3. 调用 LLM 获取回复。
  4. 如果回复包含 `<sql>` 标签，提取 SQL 并执行。
  5. 将查询结果和原始用户问题再次调用 LLM 生成最终答案。
  6. 更新对话历史，返回最终回复。

#### **代码生成要求**
请生成以下文件和代码：

1. **`main.py`**：FastAPI 应用主文件，包含 API 路由、会话管理、LLM 调用、SQL 执行逻辑。
2. **`config.py`**：配置管理（数据库连接、OpenAI GPT 模型/deepseek/千问 API 密钥、模型名称等，从环境变量读取）。
3. **`database.py`**：MySQL 连接和查询执行函数（同步或异步）。
4. **`llm.py`**：LLM 调用封装，包含构建提示词、解析回复、提取 SQL 等功能。
5. **`session.py`**：简单的内存会话存储（字典），提供获取/更新会话历史的函数。
6. **`prompts.py`**：存储系统提示词模板，包含动态插入时间和学期。
7. **`utils.py`**：辅助函数，如时间解析、节次转换等。
8. **`requirements.txt`**：项目依赖列表。

代码需包含充分的注释，解释关键步骤。使用环境变量管理敏感信息（如 `DB_HOST`、`DB_USER`、`DB_PASSWORD`、`OPENAI_API_KEY`）。假设数据库 Schema 已经存在，无需创建表。

**特别说明**：
- 在调用 LLM 时，需包含系统提示词和对话历史（用户消息和助手回复）。可以使用 OpenAI GPT 模型/deepseek/千问 的 `ChatCompletion` API，格式为 `messages` 列表。
- SQL 提取使用正则表达式，例如 `re.search(r'<sql>(.*?)</sql>', response, re.DOTALL | re.IGNORECASE)`。
- 执行 SQL 前，应验证是否为 SELECT 语句（简单检查是否以 `SELECT` 开头，忽略大小写），并设置 `cursor.execute(sql, params=None)` 避免拼接。
- 查询结果转换为 JSON 可序列化格式（如列表字典），便于传递给 LLM 生成最终回答。
- 为防止无限循环，设定最多两次 LLM 调用（即一次 SQL 生成 + 一次最终回答）。
